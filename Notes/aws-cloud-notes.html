📅 Day 1 - Module 1: Introduction to Cloud Computing

1.1 What is Cloud Computing?

Cloud Computing is the on-demand delivery of IT resources over the Internet with pay-as-you-go pricing. Instead of buying, owning, and maintaining physical data centers and servers, you can access technology services on an as-needed basis.

Simple Analogy:

Think of cloud computing like electricity in your home. You don't own a power plant; you just plug in and pay for what you use. Similarly, with cloud computing, you don't need to own servers—you just use AWS resources and pay for what you consume.

🌐 On-Demand Access

Get resources when you need them, instantly

💳 Pay-as-You-Go

Only pay for what you actually use

📈 Scalability

Scale up or down based on demand

🌍 Global Reach

Deploy applications worldwide in minutes

1.3 Benefits of Cloud Computing

📊 Scalability

The ability to accommodate larger loads by making hardware stronger (scale up) or adding more nodes (scale out).

Example:

An e-commerce website during Black Friday can automatically add more servers to handle increased traffic, then scale down after the rush.

🔄 Elasticity

Once a system is scalable, elasticity means it can scale automatically based on load.

Example:

AWS Auto Scaling automatically adds EC2 instances when CPU usage exceeds 70% and removes them when usage drops below 30%.

💰 Pay-as-You-Go Pricing

No upfront costs or long-term commitments. Pay only for the compute power, storage, and other resources you use.

Example:

Instead of buying a $10,000 server that might sit idle 50% of the time, you pay AWS only for the hours you actually use the compute resources.

⚡ Speed & Agility

Deploy resources in minutes, not weeks

🌍 Global in Minutes

Deploy to multiple regions worldwide quickly

💡 Innovation

Focus on business, not infrastructure

1.4 Shared Responsibility Model

Security and compliance is a shared responsibility between AWS and the customer. AWS is responsible for security OF the cloud, while customers are responsible for security IN the cloud.

AWS Responsibility

"Security OF the Cloud"

  • Physical data center security
  • Hardware infrastructure
  • Network infrastructure
  • Virtualization layer
  • Managed services (RDS, S3)

Customer Responsibility

"Security IN the Cloud"

  • Customer data
  • Applications
  • Identity & Access Management
  • Operating system patches
  • Firewall configuration
  • Encryption
Example: EC2 Instance Security
  • AWS secures: The physical server, hypervisor, network infrastructure
  • You secure: The operating system, installed applications, security groups, data encryption, IAM policies
Important: Even though AWS secures the infrastructure, if you misconfigure an S3 bucket to be publicly accessible with sensitive data, that's YOUR responsibility!

1.5 Reflection Questions

  1. Q: How does cloud computing differ from traditional on-premises infrastructure?
    A: Traditional infrastructure requires upfront capital investment, physical space, maintenance staff, and capacity planning. Cloud computing offers on-demand resources, pay-as-you-go pricing, no maintenance overhead, and instant global deployment.
  2. Q: Which service model (IaaS, PaaS, SaaS) do you think most companies adopt first, and why?
    A: Most companies start with SaaS (like email, CRM) because it requires zero infrastructure management. For development, they often move to IaaS (EC2) for maximum control, then gradually adopt PaaS for faster deployment.
  3. Q: In the Shared Responsibility Model, what's one example of a security task AWS handles vs one you must handle?
    A: AWS handles physical data center security (guards, cameras, access control). You must handle IAM policies to control who can access your resources.

2.2 AWS Regions

What is a Region?

A Region is a physical location around the world where AWS clusters data centers. Each Region consists of multiple, isolated Availability Zones.

Region Naming Convention:

  • us-east-1 → US East (N. Virginia)
  • eu-west-1 → Europe (Ireland)
  • ap-southeast-1 → Asia Pacific (Singapore)
  • sa-east-1 → South America (São Paulo)

How to Choose a Region?

  1. Compliance & Data Residency: Some countries require data to stay within borders (e.g., GDPR in Europe)
  2. Latency: Choose a region close to your users for faster response times
  3. Service Availability: Not all AWS services are available in all regions
  4. Pricing: Costs vary by region (us-east-1 is often cheapest)
Example: Region Selection

Scenario: You're building an app for European customers with strict GDPR compliance.

  • ✅ Choose: eu-west-1 (Ireland) or eu-central-1 (Frankfurt)
  • ❌ Avoid: us-east-1 (Virginia) - data residency issues
  • 📊 Benefit: Lower latency for EU users + GDPR compliance

2.3 Availability Zones (AZs)

An Availability Zone is one or more discrete data centers with redundant power, networking, and connectivity in an AWS Region. Each AZ is isolated but connected via high-bandwidth, low-latency networking.

Key Points:

  • Each Region has a minimum of 3 AZs (most have 3-6 AZs)
  • AZs are physically separated (tens of miles apart) to protect against disasters
  • Connected via private fiber-optic networks
  • Independent power, cooling, and physical security
Example: High Availability Architecture
Region: us-east-1 (N. Virginia)
├── AZ: us-east-1a
│   ├── EC2 Instance 1 (Web Server)
│   └── RDS Database (Primary)
│
├── AZ: us-east-1b
│   ├── EC2 Instance 2 (Web Server)
│   └── RDS Database (Standby Replica)
│
└── AZ: us-east-1c
    └── EC2 Instance 3 (Web Server)

If one AZ fails, your app continues running in other AZs!
Best Practice: Always deploy your application across at least 2 Availability Zones for high availability and fault tolerance!

2.4 Edge Locations & CloudFront

Edge Locations are AWS data centers designed to deliver content to end users with lower latency. Used by Amazon CloudFront (AWS's Content Delivery Network).

How CloudFront Works

  1. User in Tokyo requests a video from your website
  2. Request goes to nearest Edge Location (Tokyo)
  3. If content is cached, Edge Location serves it immediately (fast!)
  4. If not cached, Edge Location fetches from origin (S3/EC2), caches it, then serves
  5. Next user in Tokyo gets cached content instantly
Example: Without vs With CloudFront
Scenario Without CloudFront With CloudFront
User Location Tokyo, Japan Tokyo, Japan
Server Location us-east-1 (Virginia) Origin in us-east-1, Edge in Tokyo
Latency ~150ms ~10ms (from Edge cache)
User Experience Slow load times Fast, responsive

2.5 Reflection Questions

  1. Q: If your users are in Europe but your servers are in the US, what challenges might arise?
    A: High latency (slow page loads), potential data residency/GDPR compliance issues, increased network costs, poor user experience.
  2. Q: Why might a company choose to deploy in multiple Availability Zones?
    A: High availability - if one AZ fails (power outage, network issue), the application continues running in other AZs. Provides fault tolerance and disaster recovery.
  3. Q: How do edge locations improve user experience?
    A: Edge locations cache content closer to users, dramatically reducing latency. A user in Sydney gets content from Sydney edge location instead of waiting for data from Virginia.

3.1 Compute Services

☁️ Amazon EC2 (Elastic Compute Cloud)

Virtual servers in the cloud. Think of EC2 as renting a computer in AWS's data center that you can configure and control remotely.

Key Concepts:

  • Instance: A virtual server
  • Instance Types: Different combinations of CPU, memory, storage (t2.micro, m5.large, etc.)
  • AMI (Amazon Machine Image): Template with OS and software
  • EBS (Elastic Block Store): Persistent storage for EC2
Example: Launching an EC2 Instance (AWS Console)
  1. Go to EC2 Dashboard → Click "Launch Instance"
  2. Choose AMI (Amazon Linux 2, Ubuntu, Windows, etc.)
  3. Choose Instance Type (t2.micro for free tier)
  4. Configure network settings (VPC, subnet, security group)
  5. Add storage (default 8GB EBS volume)
  6. Review and Launch

⚡ AWS Lambda

Serverless compute service - run code without managing servers. You pay only for the compute time you consume (charged per millisecond).

When to Use Lambda:

  • Processing file uploads (resize images, extract metadata)

  • Scheduled tasks (cron jobs)
  • Real-time stream processing
  • Event-driven applications

3.2 Storage Services

📦 Amazon S3 (Simple Storage Service)

Object storage service offering industry-leading scalability, data availability, security, and performance. Store and retrieve any amount of data from anywhere.

Key Concepts:

  • Bucket: Container for objects (like a folder)
  • Object: Files stored in S3 (up to 5TB each)
  • Key: Unique identifier/path for an object
  • Durability: 99.999999999% (11 nines) - designed to sustain loss of data in 2 facilities
  • Availability: 99.99% - accessible when you need it
Example: S3 Use Cases
  • 🖼️ Media Storage: Store images, videos for websites/apps
  • 💾 Backup & Restore: Database backups, file archives
  • 🌐 Static Website Hosting: Host HTML/CSS/JS files
  • 📊 Data Lakes: Store massive amounts of data for analytics
  • 📱 Mobile/App Storage: User uploads, app assets

3.3 Networking Services

🌐 Amazon VPC (Virtual Private Cloud)

A logically isolated section of the AWS Cloud where you can launch AWS resources in a virtual network that you define. Think of it as your own private network in AWS.

VPC Components:

  • Subnets: Segments of VPC IP address range (public/private)
  • Internet Gateway (IGW): Allows communication with internet
  • Route Tables: Rules for directing network traffic
  • NAT Gateway: Allows private subnets to access internet
  • Network ACLs: Firewall at subnet level
  • Security Groups: Firewall at instance level
Example: VPC Architecture
VPC (10.0.0.0/16)
│
├── Public Subnet (10.0.1.0/24)
│   ├── Web Server EC2 (10.0.1.10)
│   ├── Load Balancer
│   └── Internet Gateway → Access from internet
│
└── Private Subnet (10.0.2.0/24)
    ├── Application Server EC2 (10.0.2.10)
    ├── Database (RDS)
    └── NAT Gateway → Access to internet (outbound only)

Public Subnet: Can receive traffic from internet (web servers)
Private Subnet: No direct internet access (databases, app servers)

🔒 Security Groups

Virtual firewalls that control inbound and outbound traffic for EC2 instances. Security groups are stateful (if you allow inbound, response is automatically allowed).

Example: Web Server Security Group
Type Protocol Port Source Description
Inbound HTTP 80 0.0.0.0/0 Allow web traffic from anywhere
Inbound HTTPS 443 0.0.0.0/0 Allow secure web traffic
Inbound SSH 22 Your IP/32 Allow SSH only from your IP
Outbound All All 0.0.0.0/0 Allow all outbound traffic 
# Create security group
aws ec2 create-security-group \
  --group-name WebServerSG \
  --description "Security group for web servers" \
  --vpc-id vpc-0123456789abcdef0

# Add inbound rule for HTTP
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 80 \

# Add inbound rule for SSH (your IP only)
aws ec2 authorize-security-group-ingress \

3.4 Reflection Questions

  1. Q: When would you choose EC2 over Lambda for running code?
    A: Choose EC2 for: long-running applications, applications needing persistent connections, custom OS/software requirements, execution time > 15 minutes, applications with consistent traffic. Choose Lambda for: short tasks, event-driven workloads, variable/unpredictable traffic, serverless architecture.
  2. Q: Why is S3 considered "infinitely scalable" storage?
    A: S3 automatically scales to handle any amount of data without you managing capacity. No need to provision storage in advance - just upload files and S3 handles the rest. It's designed to store trillions of objects.
  3. Q: How do security groups differ from IAM policies in controlling access?
    A: Security Groups control network access (who can connect to your EC2 instance - IP addresses, ports). IAM policies control API access (what actions users/services can perform on AWS resources - create, delete, read).

4.4 IAM Roles

An IAM Role is an identity with permissions that determine what the identity can and cannot do. Unlike users, roles don't have long-term credentials. They're assumed temporarily by users, applications, or AWS services.

When to Use Roles:

  • Give EC2 instances permission to access S3 or RDS
  • Allow Lambda functions to write logs to CloudWatch
  • Enable cross-account access (Account A access Account B's resources)
  • Grant temporary access to users from another identity provider
Example: EC2 Instance Role

Scenario: You have an EC2 instance that needs to upload files to S3.

❌ Bad Practice: Store AWS access keys on the EC2 instance

2. Attach Policy: AmazonS3FullAccess (or custom S3 write policy)
3. Attach Role to EC2 Instance
4. EC2 instance can now access S3 without hardcoded credentials!

4.8 Reflection Questions

  1. Q: Why is it dangerous to use the root account for daily tasks?
    A: The root account has unrestricted access to everything - billing, all services, account closure. If credentials are compromised, an attacker could delete your entire infrastructure, rack up huge bills, or lock you out. Plus, actions can't be easily tracked to individuals.
  2. Q: What's the difference between a role and a user in IAM?
    A: Users have long-term credentials (username/password, access keys) and represent people or applications. Roles have no credentials - they're assumed temporarily by users, applications, or AWS services. Roles are more secure for applications because credentials rotate automatically.
  3. Q: How would you explain "least privilege" to a non-technical stakeholder?
    A: It's like giving an employee only the keys they need. A janitor doesn't need keys to the CEO's office or the vault. Similarly, a developer shouldn't have access to delete production databases or modify billing. Give people only what they need to do their job - no more, no less.

1.1 Auto Scaling Groups (ASG)

Auto Scaling Groups automatically adjust the number of EC2 instances in your application based on demand. Scale out (add instances) during high traffic, scale in (remove instances) during low traffic.

Benefits of Auto Scaling:

  • 💰 Cost Optimization: Only pay for instances you need

  • 🔄 High Availability: Replace unhealthy instances automatically
  • ⚖️ Load Distribution: Distribute traffic across multiple instances

├── Desired: 2 instances
└── Maximum: 10 instances

Black Friday Sale (Peak Traffic):
├── CPU Usage exceeds 70%
├── Auto Scaling triggers: Add 3 instances
├── New Total: 5 instances running
└── Traffic handled smoothly

After Sale (Low Traffic):
├── CPU Usage drops below 30%
├── Auto Scaling triggers: Remove 3 instances
├── Back to: 2 instances running
└── Cost optimized!

Ultra-high performance, operates at Layer 4 (transport layer)

  • Ultra-low latency (microseconds)
  • Static IP address support
  • Preserves source IP address
  • Best for TCP/UDP traffic, gaming, IoT

🔧 Gateway Load Balancer (GWLB)

Deploy, scale, and manage third-party virtual appliances

Use Cases:

  • Firewalls
  • Intrusion detection/prevention systems

1.5 Reflection Questions

  1. Q: Why is elasticity one of the biggest advantages of cloud?
    A: Elasticity allows you to match resources to demand automatically. Without cloud, you'd buy servers for peak load (expensive, mostly idle). Or worse, under-provision and crash during spikes. Cloud elasticity = save money during low traffic, handle spikes during high traffic, all automatically.
  2. Q: How does an ALB improve availability compared to a single EC2 instance?
    A: Single instance = single point of failure (if it crashes, site is down). ALB distributes traffic across multiple instances in multiple AZs. If one instance fails, ALB routes traffic to healthy instances. If an entire AZ fails, instances in other AZs keep running.
  3. Q: Can you think of a scenario where scaling down is just as important as scaling up?
    A: A news website might scale up during breaking news (millions of readers), but needs to scale down after the story dies to avoid paying for idle servers. E-commerce scales up during sales, down overnight. Development environments can scale to zero outside business hours.

2.3 Amazon DynamoDB

Amazon DynamoDB is a fully managed NoSQL database service that provides fast and predictable performance with seamless scalability. Single-digit millisecond latency at any scale.

Key Features:

  • ⚡ Performance: Single-digit millisecond response time
  • 📈 Scalability: Automatically scales up/down based on traffic
  • 🔒 Fully Managed: No servers to manage, automatic backups
  • 🌍 Global Tables: Multi-region, fully replicated tables
  • 💰 Pay-per-Request: Or provision capacity for predictable workloads
  • 📊 Streams: Capture table changes for real-time processing

DynamoDB Core Concepts

  • Table: Collection of data (like a table in SQL, but schemaless)
  • Item: A single data record (like a row in SQL)
  • Attribute: A data element in an item (like a column in SQL)
  • Primary Key: Uniquely identifies each item (Partition Key or Partition Key + Sort Key)
  • Secondary Index: Query data using alternate keys

When to Use DynamoDB vs RDS

✅ Choose DynamoDB when:

  • You need single-digit millisecond latency at any scale
  • Your data access patterns are simple (key-value lookups)
  • You need to scale massively (millions of requests/sec)
  • Your schema is flexible or frequently changing
  • Building serverless applications
  • Examples: Session storage, gaming leaderboards, IoT data, mobile apps

✅ Choose RDS when:

  • You need complex queries (JOINs, aggregations, GROUP BY)
  • You require strong ACID transactions
  • Your data has complex relationships
  • You're migrating an existing relational database
  • You need SQL compatibility
  • Examples: E-commerce orders, financial transactions, CRM, ERP

2.4 Practical Example: Choosing the Right Database

Scenario 1: E-commerce Platform

Requirements:

  • Store product catalog, orders, customers
  • Complex queries: "Show all orders for customer X in the last 30 days where total > $100"
  • Need transactions: Order must decrease inventory and charge customer atomically
  • Existing SQL-based application

✅ Solution: Amazon RDS (MySQL or PostgreSQL)

Reason: Complex queries, relationships, transactions, SQL compatibility

Scenario 2: Gaming Leaderboard

Requirements:

  • Store player scores (millions of players)
  • Simple queries: "Get score for player X" or "Get top 10 players"
  • Need ultra-low latency (< 10ms)
  • Traffic spikes during game launches
  • Global player base

✅ Solution: Amazon DynamoDB with Global Tables

Reason: Simple access patterns, low latency, massive scale, global replication

Scenario 3: IoT Sensor Data

Requirements:

  • Store data from millions of sensors
  • Each sensor sends data every second
  • Simple queries: "Get latest data from sensor X"
  • Retention: Keep data for 30 days, then archive
  • Need to scale automatically

✅ Solution: Amazon DynamoDB with TTL (Time To Live)

Reason: High write throughput, simple queries, automatic scaling, TTL for automatic deletion

2.5 Reflection Questions

  1. Q: What are the trade-offs between relational and NoSQL databases?
    A: Relational: Structured data, complex queries, ACID transactions, harder to scale horizontally. NoSQL: Flexible schema, simple queries, massive scale, eventually consistent, limited transaction support. Choose based on your specific needs.
  2. Q: Why might a startup choose DynamoDB over RDS?
    A: Startups value speed and don't know future scale. DynamoDB: serverless (no capacity planning), scales automatically, pay-per-request pricing (cost-effective at low volume), extremely fast development. Perfect for MVP and uncertain growth.
  3. Q: How does AWS handle backups differently in RDS vs DynamoDB?
    A: RDS: Automated daily backups to S3, point-in-time recovery, manual snapshots, retention up to 35 days. DynamoDB: Continuous backups with point-in-time recovery (PITR), on-demand backups, no retention limit, faster restore times.

3.6 Reflection Questions

  1. Q: Why is monitoring critical even for small applications?
    A: Without monitoring, you're flying blind. You won't know when things break until customers complain. Even small apps need to know: Is it running? Is it slow? Are there errors? Monitoring enables proactive problem-solving instead of reactive firefighting. Plus, it helps you understand usage patterns and optimize costs.
  2. Q: How would you use CloudWatch alarms to prevent downtime?
    A: Set alarms for critical metrics: high CPU (triggers Auto Scaling before crash), low disk space (add storage before it fills), high error rates (alert team before users are affected), unhealthy instances (replace before service degrades). Alarms let you fix problems before they cause downtime.
  3. Q: What kind of suspicious activity might CloudTrail help you detect?
    A: Unauthorized access (login from unusual location), privilege escalation (user giving themselves admin rights), data exfiltration (large S3 downloads), resource deletion (who terminated the production database), security group changes (opening port 22 to 0.0.0.0/0), credential usage (access keys used from multiple IPs).

4.1 AWS Pricing Models

AWS offers flexible pricing models to match your workload patterns and optimize costs. Understanding these models is crucial for cost-effective cloud architecture.

💳 On-Demand Instances

Pay for compute capacity by the hour or second with no long-term commitments.

Characteristics:

  • 💰 Pricing: Highest per-hour cost
  • ⏱️ Commitment: None - start/stop anytime
  • 📊 Best For: Unpredictable workloads, development/testing, short-term projects
  • ✅ Pros: Maximum flexibility, no upfront costs
  • ❌ Cons: Most expensive option for steady workloads
Example: t3.medium On-Demand

$0.0416/hour × 730 hours/month = $30.37/month

📅 Reserved Instances (RI)

Commit to using EC2 for 1 or 3 years in exchange for significant discounts (up to 75% off On-Demand).

Characteristics:

  • 💰 Savings: 40-75% discount vs On-Demand
  • ⏱️ Commitment: 1 year or 3 years
  • 💳 Payment: All Upfront, Partial Upfront, or No Upfront
  • 📊 Best For: Steady-state workloads (databases, always-on web servers)
  • ✅ Pros: Significant cost savings
  • ❌ Cons: Locked into commitment, less flexibility
Example: t3.medium Reserved (1-year, All Upfront)

$182 upfront = $15.17/month (50% savings vs On-Demand)

🎯 Spot Instances

Bid on unused EC2 capacity for up to 90% discount. AWS can reclaim instances with 2-minute warning.

Characteristics:

  • 💰 Savings: Up to 90% discount vs On-Demand
  • ⚠️ Risk: Can be terminated by AWS anytime
  • 📊 Best For: Fault-tolerant workloads (batch processing, data analysis, testing)
  • ✅ Pros: Massive cost savings
  • ❌ Cons: Can be interrupted, not suitable for critical workloads
Example: t3.medium Spot

~$0.0125/hour × 730 hours = $9.13/month (70% savings vs On-Demand)

Note: Spot prices fluctuate based on supply and demand

💾 Savings Plans

Flexible pricing model offering lower prices in exchange for commitment to consistent usage ($/hour) for 1 or 3 years.

Types:

  • Compute Savings Plans: Most flexible, applies to EC2, Fargate, Lambda (up to 66% savings)
  • EC2 Instance Savings Plans: Applies to specific instance family in a region (up to 72% savings)

4.5 AWS Well-Architected Framework

The AWS Well-Architected Framework helps you understand the pros and cons of decisions you make while building systems on AWS. It provides best practices across five pillars.

1️⃣ Operational Excellence

Run and monitor systems to deliver business value and continually improve processes.

Key Principles:

  • Perform operations as code (Infrastructure as Code)
  • Make frequent, small, reversible changes
  • Refine operations procedures frequently
  • Anticipate failure and learn from failures
  • Example: Use CloudFormation for infrastructure, implement CI/CD pipelines

2️⃣ Security

Protect information, systems, and assets while delivering business value.

Key Principles:

  • Implement strong identity foundation (IAM, MFA, least privilege)
  • Enable traceability (CloudTrail, VPC Flow Logs)
  • Apply security at all layers (WAF, Security Groups, encryption)
  • Automate security best practices
  • Protect data in transit and at rest (TLS, KMS encryption)
  • Example: Enable MFA, encrypt all S3 buckets, use private subnets

3️⃣ Reliability

Ensure a workload performs its intended function correctly and consistently.

Key Principles:

  • Automatically recover from failure (health checks, Auto Scaling)
  • Test recovery procedures (chaos engineering, disaster recovery drills)
  • Scale horizontally (multiple small instances vs one large)
  • Stop guessing capacity (use Auto Scaling)
  • Manage change through automation
  • Example: Deploy across multiple AZs, implement automated backups

4️⃣ Performance Efficiency

Use computing resources efficiently to meet requirements and maintain efficiency as demand changes.

Key Principles:

  • Democratize advanced technologies (use managed services)
  • Go global in minutes (deploy to multiple regions easily)
  • Use serverless architectures (Lambda, DynamoDB)
  • Experiment more often (easy to try new instance types)
  • Mechanical sympathy (use right tool for the job)
  • Example: Use ElastiCache for caching, CloudFront for CDN, choose right database

5️⃣ Cost Optimization

Avoid unnecessary costs and achieve cost-effective results.

Key Principles:

  • Implement cloud financial management
  • Adopt a consumption model (pay only for what you use)
  • Measure overall efficiency (cost per transaction, cost per user)
  • Stop spending money on undifferentiated heavy lifting (use managed services)
  • Analyze and attribute expenditure (cost allocation tags, Cost Explorer)
  • Example: Use Auto Scaling, buy Reserved Instances, implement S3 lifecycle policies
Example: Well-Architected Review Questions

Operational Excellence: Do you have runbooks for common operational tasks?

Security: Are all S3 buckets encrypted and non-public?

Reliability: Can your application survive the loss of an Availability Zone?

Performance: Have you tested different instance types to find the optimal cost/performance?

Cost Optimization: Are you using Reserved Instances for steady-state workloads?

4.6 Reflection Questions

  1. Q: Why might a company choose Reserved Instances over On-Demand?
    A: If you have predictable, steady-state workloads (production databases, always-on applications), Reserved Instances offer 40-75% savings. The commitment (1-3 years) is acceptable because you know you'll need that capacity anyway. For example, your production database will always run 24/7, so why pay On-Demand rates?
  2. Q: How can poor architecture decisions lead to higher costs?
    A: Examples: Using oversized instances (waste), not implementing Auto Scaling (paying for idle capacity), keeping all data in S3 Standard (not using lifecycle policies), deploying in expensive regions, not using CloudFront (high data transfer costs), running dev/test 24/7 instead of scheduling, not using Spot for batch jobs.
  3. Q: Which of the Well-Architected pillars do you think is most often overlooked, and why?
    A: Operational Excellence is often overlooked because teams rush to build features. They skip documentation, runbooks, and automation. This creates technical debt. Security is second - teams focus on speed over security until a breach happens. Both require upfront investment but prevent costly problems later.
