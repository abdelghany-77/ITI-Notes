Introduction to Version Control

🎯 What is Version Control?

A system that records changes to a file or set of files over time so that you can recall specific versions later. Essential for collaboration, tracking history, and managing different lines of development.

Why Use Version Control?

1
Collaboration

Multiple developers can work on the same project without overwriting each other's work

2
History Tracking

Detailed record of every change, who made it, when, and why

3
Rollbacks

Easily revert to previous stable versions of the code

4
Branching & Merging

Parallel development and integration of features

5
Experimentation

Safely try new ideas without affecting main codebase

🔥 What is Git?

A distributed version control system that tracks changes to files. Unlike centralized systems, every developer has a full copy of the repository, including its entire history. This allows for offline work and resilience.

Key Git Terminology

📁 Repository (Repo)

A folder where Git tracks changes. Contains all project files and entire history of commits.

📸 Commit

A snapshot of changes at a specific time. Each has a unique ID (hash) and commit message.

🌿 Branch

A parallel line of development. Allows work on features in isolation from main codebase.

🔗 Merge

Combining changes from different branches into one.

🌐 Remote

A hosted repository (GitHub, GitLab) where team's code is stored and shared.

👉 HEAD

A pointer to the tip of the current branch (latest commit).

📂 Working Directory

The actual files you see and edit on your local machine.

🎭 Staging Area (Index)

Intermediate area where you prepare changes for next commit.

Git Basic Commands

Setup & Configuration

git init <directory>
Create an empty Git repository in specified directory
git clone <repo_url>
Clone a repository from remote location to local machine
git config --global user.name "<name>"
Set global username for all repositories
git config --global user.email "<email>"
Set global email for all repositories
git config --global alias.<alias-name> "<git-command>"
Create shortcuts for Git commands

Basic Workflow

git add <file_or_directory>
Stage changes for next commit
git commit -m "<message>"
Commit staged snapshot with message
git status
Show which files are staged, unstaged, and untracked
git diff
Show unstaged changes between working directory and staging area
git diff --cached
Show staged changes compared to last commit

History & Viewing Changes

git log
Display entire commit history
git log --oneline
Condense each commit to single line
git log --graph --decorate
Show text-based graph with branch/tag names
git log --author="<pattern>"
Search commits by specific author
git log --grep="<pattern>"
Search commits by commit message

Undoing Changes

⚠️ Important Warning

Some Git operations are destructive and cannot be undone. Always be careful with reset --hard and similar commands!

git revert <commit>
Create new commit that undoes changes (safe, preserves history)
git reset <file>
Remove file from staging area (unstage)
git reset --hard
⚠️ DESTRUCTIVE: Reset staging and working directory to last commit
git clean -n
Show which untracked files would be removed (dry run)
git clean -f
Remove untracked files from working directory

Merging & Rebasing

🔀 Merging Process

  1. Switch to target branch: git checkout main
  2. Merge feature branch: git merge feature/login
  3. Resolve conflicts if any
  4. Commit the merge

⚔️ Handling Merge Conflicts

When Git cannot automatically combine changes:

  1. Open conflicted files (look for <<<<<<< HEAD markers)
  2. Manually edit to resolve conflicts
  3. Stage resolved files: git add <file>
  4. Commit the merge: git commit -m "Merge conflict resolved"
git merge <branch_name>
Merge specified branch into current branch
git rebase <base>
Rebase current branch onto specified base (creates linear history)
git rebase -i <base>
Interactive rebase (squash, reorder, edit commits)

Remote Repositories

git remote add origin <github-repo-url>
Connect local repo to remote repository
git push -u origin main
Push main branch and set upstream tracking
git fetch <remote>
Download changes from remote (doesn't merge)
git pull <remote>
Fetch and merge remote changes (git fetch + git merge)
git push <remote> <branch>
Push local branch to remote repository
git push --force
⚠️ Force push (rewrites remote history - use with caution!)

Best Practices

💬 Commit Message Conventions

  • Use imperative tense: "Fix login bug" (not "Fixed login bug")
  • Keep first line under 50 characters
  • Use conventional commits: feat:, fix:, docs:, style:, refactor:
  • Example: feat: Add user authentication system

🌿 Branch Naming Conventions

  • feature/user-authentication
  • bugfix/navbar-alignment
  • hotfix/security-patch
  • docs/api-documentation
🔒 Security

Never commit secrets, passwords, or API keys. Use .gitignore and environment variables.

👥 Code Reviews

Use Pull Requests for collaboration and enforce reviews before merging.

⚛️ Atomic Commits

Make small, focused commits that address single logical changes.

🔄 Regular Pulls

Frequently pull changes from remote to minimize merge conflicts.

GitHub Security

SSH Keys Setup

ssh-keygen -t ed25519 -C "your-email@example.com"
Generate new SSH key pair
cat ~/.ssh/id_ed25519.pub
Display public key to add to GitHub

🔑 SSH Key Benefits

  • No need to enter password for every operation
  • More secure than HTTPS with passwords
  • Required for many Git operations in enterprise

Branch Protection Rules

🛡️ Recommended Protection Settings

  • Require PR reviews before merging
  • Require status checks to pass
  • Require branches to be up to date
  • Restrict pushes to matching branches
  • Require administrator review for high-risk changes

Interview Questions & Answers

💼 Common Git Interview Questions

Q: What's the difference between git fetch and git pull?
A: git fetch downloads changes from remote but doesn't merge them. git pull is git fetch + git merge combined.
Q: How do you revert a commit?
A: Use git revert <commit-hash> to create a new "undo" commit (safe), or git reset --hard HEAD~1 to remove completely (destructive).
Q: What is a .gitignore file?
A: A file specifying which files/directories Git should ignore (e.g., node_modules/, .env, build files).
Q: Explain the difference between merge and rebase.
A: Merge combines branches and preserves history. Rebase moves commits to create linear history by replaying changes.
Q: What are the three states of files in Git?
A: Modified (changed but not staged), Staged (marked for next commit), Committed (stored in repository).
Q: How do you undo the last commit but keep changes?
A: Use git reset --soft HEAD~1 to undo commit but keep changes staged, or git reset HEAD~1 to unstage them too.
Q: What is a fast-forward merge?
A: When the target branch hasn't diverged, Git simply moves the pointer forward. No merge commit is created.
Q: How do you resolve merge conflicts?
A: Open conflicted files, edit to resolve conflicts (remove <<<, ===, >>> markers), stage files with git add, then commit.
