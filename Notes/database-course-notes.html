Database Introduction & Concepts

File-Based System Disadvantages

üö´ Major Problems with File-Based Systems

  • Separation & Isolation: Each user has separate copies leading to inconsistencies
  • Incompatible Formats: Different programs require specific file formats
  • Program-Data Dependence: Applications need metadata management code
  • Data Redundancy: Multiple copies of same data causing inconsistencies
  • No Database Integrity: Limited data sharing and no inherent integrity
  • Lengthy Development: Programmers handle file formats and metadata

Basic Definitions

üóÉÔ∏è Database

A collection of related data organized for efficient storage and retrieval

üõ†Ô∏è DBMS

Database Management System - software that facilitates creation and maintenance of databases

üèóÔ∏è Database System

Complete system including DBMS software, data, applications, and users

DBMS Advantages

‚úÖ Benefits of Database Management Systems

  • Data Sharing: Multiple users can access different views of data
  • Integrity Constraints: Enforced data validation rules
  • Improved Data Quality: Consistency through centralized control
  • Access Control: Security features for unauthorized access prevention
  • Backup & Recovery: Built-in disaster recovery mechanisms
  • Minimal Redundancy: Increased integrity and consistency
  • Program-Data Independence: Applications don't worry about data formats
  • Standardization: SQL standard for better accessibility

‚ö†Ô∏è DBMS Disadvantages

  • Requires specialized expertise to use effectively
  • DBMS software can be expensive
  • May have compatibility issues with other systems

Entity-Relationship Modeling (ERD)

üîó What is ERD?

Entity-Relationship Diagram (ERD) identifies information required by business by displaying relevant entities and relationships. It's a visual representation of database structure showing how data elements relate to each other.

Basic Constructs of E-R Model

üì¶ Entities

Person, place, object, event, concept

Represented by rectangles
üè∑Ô∏è Attributes

Properties or characteristics

Represented by ellipses
üîó Relationships

Links between entities

Represented by diamonds

Strong vs Weak Entities

üí™ Strong Entity

Has a primary key that uniquely identifies each instance

  • Independent existence
  • Has complete primary key
  • Represented by single-line rectangle
üîó Weak Entity

Doesn't have sufficient attributes to form a primary key

  • Depends on strong entity
  • Uses partial key + owner's key
  • Represented by double-line rectangle

Types of Attributes

üî∏ Simple Attribute

Cannot be broken down further

Example: cust-id, cust-name

üîπ Composite Attribute

Composed of several other attributes

Example: Name (First, Last), Address (Street, City, State)

üìä Derived Attribute

Value calculated from other attributes

Example: Age from birth date, Net Salary from Salary - Deduction

üîÑ Multi-valued Attribute

Can have multiple values for single entity

Example: Phone numbers, Email addresses

Relationship Properties

üî¢ Degree of Relationships

  • Unary (Recursive): Between instances of one entity (e.g., Employee manages Employee)
  • Binary: Between instances of two entities (most common)
  • Ternary: Among instances of three entities
1:1 One-to-One

Each instance relates to exactly one instance in another entity

1:N One-to-Many

One instance relates to multiple instances in another entity

M:N Many-to-Many

Multiple instances relate to multiple instances

üìã Participation Constraints

  • Total Participation: Entity instance MUST participate (double line)
  • Partial Participation: Entity instance MAY participate (single line)

Keys in Database Design

üîë Primary Key

Chosen unique identifier, must be unique & not null

üóùÔ∏è Candidate Key

Set of attributes that can uniquely identify entity

üîó Foreign Key

Attribute referring to primary key in another table

üîß Composite Key

Primary key composed of two or more attributes

‚ö° Super Key

Set of attributes that uniquely identifies entity (superset of candidate key)

üîÑ Alternate Key

Candidate key not chosen as primary key

ER-to-Relational Mapping

üîÑ Mapping Process

Converting Entity-Relationship diagrams into relational database tables following systematic steps to preserve data integrity and relationships.

Mapping Steps

Regular Entity Types
  • Create table for each regular entity
  • Choose primary key from key attributes
  • Include simple components of composite attributes
  • Create separate table for multivalued attributes
  • Usually ignore derived attributes
Weak Entity Types
  • Create table for each weak entity
  • Add foreign key from owner entity
  • Primary key = partial key + owner's primary key
Binary 1:1 Relationships
  • Both mandatory: Merge into single table
  • One optional: Add foreign key to mandatory side
  • Both optional: Create separate relationship table
Binary 1:N Relationships
  • Add foreign key to "N-side" table
  • Reference primary key of "1-side" table
  • Include relationship attributes in N-side table
Binary M:N Relationships
  • Create new relationship table
  • Add foreign keys from both parent tables
  • Include relationship attributes
  • Primary key = combination of foreign keys
Unary Relationships
  • Add foreign key to same table
  • Reference own primary key
  • Example: Manager_ID in Employee table

DDL - Data Definition Language

üèóÔ∏è CREATE TABLE
CREATE TABLE Employee ( ID int PRIMARY KEY, FName char(50) NOT NULL, LName char(50), Address char(50), Birth_Date date );
üóëÔ∏è DROP TABLE
DROP TABLE Employee;
üîß ALTER TABLE
-- Add column ALTER TABLE employee ADD City varchar(30); -- Drop column ALTER TABLE employee DROP City;

DML - Data Manipulation Language

‚ûï INSERT
-- Single row INSERT INTO Employee VALUES (1, 'John', 'Doe', '123 Main St', '1990-01-01'); -- Multiple rows INSERT INTO Employee VALUES (2, 'Jane', 'Smith', '456 Oak Ave', '1985-05-15'), (3, 'Bob', 'Johnson', '789 Pine St', '1992-12-20');
‚úèÔ∏è UPDATE
-- Update with WHERE clause UPDATE Employee SET Address = 'New Address' WHERE ID = 1; -- Update with expression UPDATE Employee SET LName = UPPER(LName);
üóëÔ∏è DELETE
-- Delete specific rows DELETE FROM Employee WHERE ID = 1; -- Delete all rows DELETE FROM Employee;
üîÑ TRUNCATE
-- Faster than DELETE, cannot be used with FOREIGN KEY constraints TRUNCATE TABLE Employee;

DQL - SELECT Statement

üìä SELECT Processing Order

  1. FROM: Identifies involved tables
  2. WHERE: Filters rows based on conditions
  3. GROUP BY: Groups rows by specified columns
  4. HAVING: Filters groups based on conditions
  5. SELECT: Specifies columns to return
  6. ORDER BY: Sorts the result set
üîç Basic SELECT
-- All columns SELECT * FROM Employee; -- Specific columns SELECT FName, LName FROM Employee; -- With aliases SELECT FName AS 'First Name', LName AS 'Last Name' FROM Employee;
üéØ WHERE Clause
-- Comparison operators SELECT * FROM Employee WHERE ID > 5; -- String matching SELECT * FROM Employee WHERE FName LIKE 'J%'; -- Multiple conditions SELECT * FROM Employee WHERE ID BETWEEN 1 AND 10 AND FName IN ('John', 'Jane');
üî¢ NULL Handling
-- Check for NULL SELECT * FROM Employee WHERE Address IS NULL; -- Handle NULL values SELECT ISNULL(Address, 'No Address') FROM Employee; -- COALESCE function SELECT COALESCE(Address, City, 'Unknown') FROM Employee;
üìã Sorting & Distinct
-- Sort results SELECT * FROM Employee ORDER BY LName, FName; -- Remove duplicates SELECT DISTINCT City FROM Employee; -- Limit results SELECT TOP 10 * FROM Employee;

Joins

üîó INNER JOIN
SELECT e.FName, d.DeptName FROM Employee e INNER JOIN Department d ON e.DeptID = d.DeptID;
‚óÄÔ∏è LEFT OUTER JOIN
SELECT e.FName, d.DeptName FROM Employee e LEFT JOIN Department d ON e.DeptID = d.DeptID;
‚ñ∂Ô∏è RIGHT OUTER JOIN
SELECT e.FName, d.DeptName FROM Employee e RIGHT JOIN Department d ON e.DeptID = d.DeptID;
üåê FULL OUTER JOIN
SELECT e.FName, d.DeptName FROM Employee e FULL OUTER JOIN Department d ON e.DeptID = d.DeptID;
‚ùå CROSS JOIN
-- Cartesian product SELECT e.FName, d.DeptName FROM Employee e CROSS JOIN Department d;
üîÑ SELF JOIN
-- Employee and their manager SELECT e.FName AS Employee, m.FName AS Manager FROM Employee e LEFT JOIN Employee m ON e.ManagerID = m.ID;

Subqueries

üìä Basic Subquery
-- Find department with highest paid employee SELECT dname FROM dept WHERE deptno = ( SELECT deptno FROM emp WHERE sal = (SELECT MAX(sal) FROM emp) );
üîç EXISTS Condition
-- Find suppliers with orders SELECT * FROM suppliers s WHERE EXISTS ( SELECT 1 FROM orders o WHERE s.supplier_id = o.supplier_id );
üìà ALL Keyword
-- Salary greater than all in department 5 SELECT * FROM employee WHERE salary > ALL ( SELECT salary FROM employee WHERE Dno = 5 );

Aggregate Functions

üìä Basic Aggregates
SELECT COUNT(*) AS TotalEmployees, SUM(salary) AS TotalSalary, AVG(salary) AS AvgSalary, MAX(salary) AS MaxSalary, MIN(salary) AS MinSalary FROM Employee;
üî¢ GROUP BY
-- Group by department SELECT dno, COUNT(*) AS EmpCount, AVG(salary) AS AvgSalary FROM employee GROUP BY dno;
üéØ HAVING
-- Filter groups SELECT pnumber, pname, COUNT(*) AS WorkerCount FROM project_assignments GROUP BY pnumber, pname HAVING COUNT(*) > 2;

DCL - Data Control Language

‚úÖ GRANT
-- Grant SELECT permission GRANT SELECT ON employees TO Ahmed; -- Grant ALL permissions GRANT ALL ON department TO Mary, Ahmed; -- Grant with GRANT option GRANT SELECT ON employees TO Ahmed WITH GRANT OPTION;
‚ùå REVOKE
-- Revoke specific permission REVOKE UPDATE ON department FROM Mary; -- Revoke all permissions REVOKE ALL ON department FROM Mary, Ahmed;

Database Normalization

üìê What is Normalization?

Process of structuring data to minimize duplication and inconsistencies. Involves breaking single table into multiple tables and defining relationships between them, usually done in stages with each stage applying rules to eliminate anomalies.

Goals & Anomalies

üö´ Problems to Avoid

  • Insertion Anomaly: Adding new rows forces creation of duplicate data
  • Deletion Anomaly: Deleting rows may cause loss of needed data
  • Modification Anomaly: Changes force updates to multiple rows due to duplication

Root Cause: Multiple entity types combined into one relation

Functional Dependency

üîó Functional Dependency (A ‚Üí B)

For every valid instance of A, that value uniquely determines the value of B. The existence of B depends on the value of A.

Examples: SSN ‚Üí ENAME, PNUMBER ‚Üí {PNAME, PLOCATION}

üéØ Full Functional Dependency

Attribute fully dependent on entire primary key

Example: {SID, Subject} ‚Üí Grade

üîó Partial Functional Dependency

Attribute dependent on part of composite key

Example: SID ‚Üí SName, Subject ‚Üí Teacher

üîÑ Transitive Functional Dependency

Attribute dependent on non-key attribute

Example: ZipCode ‚Üí City

Normal Forms

1Ô∏è‚É£ First Normal Form (1NF)

Goal: Remove multivalued attributes

  • No multivalued or composite attributes
  • All columns must be atomic
  • No repeating groups
  • Move repeating groups to new table with foreign key
-- Before 1NF Student(SID, SName, Subjects) Subjects: Math, Physics, Chemistry -- After 1NF Student(SID, SName, Birthdate, City, ZipCode) Student_Subject(SID, Subject, Grade, Teacher)
2Ô∏è‚É£ Second Normal Form (2NF)

Goal: Remove partial dependencies

  • Must be in 1NF
  • Every non-key attribute fully dependent on primary key
  • No partial dependencies on composite keys
-- Before 2NF (Teacher depends only on Subject) Student_Subject(SID, Subject, Grade, Teacher) -- After 2NF Student_Subject(SID, Subject, Grade) Subject(Subject, Teacher)
3Ô∏è‚É£ Third Normal Form (3NF)

Goal: Remove transitive dependencies

  • Must be in 2NF
  • No transitive dependencies (A ‚Üí B ‚Üí C)
  • Non-key attributes depend only on primary key
-- Before 3NF (ZipCode ‚Üí City is transitive) Student(SID, SName, Birthdate, City, ZipCode) -- After 3NF Student(SID, SName, Birthdate, ZipCode) City_Info(ZipCode, City)

üìã Original ITI Students Sheet (Unnormalized)

Attributes: Appno, Name, F-code, Faculty, Address, Telno, Grade, Att.Hrs, Sdate

Issues: Multivalued attributes (Telno), composite attributes, potential redundancy

üîÑ Converting to 1NF

Action: Separate multivalued telephone numbers

üìä Platform Table

üë• Students_pf Table

üìû Std_Tel Table

üîÑ Converting to 2NF

Action: Remove partial dependencies - separate student details

üë§ Students Table
Students(appno, name, faculty, FCode, address) -- Student details separated from platform relationship
üìö Students_pf Table
Students_pf(pfname, appno, grade, attd, start_date) -- Only attributes dependent on both pfname and appno
üìû Unchanged Tables
Platform(pfname, pfdesc, pfManager) Std_Tel(appno, telno) -- No changes needed
üîÑ Converting to 3NF

üè´ Fac_majors Table
Fac_majors(FCode, faculty) -- Separate table for faculty and F-Code relationship
üìû Unchanged Tables
Platform(pfname, pfdesc, pfManager) Std_Tel(appno, telno) Students_pf(pfname, appno, grade, attd, start_date) -- No further changes needed

Normalization Benefits

‚úÖ Advantages of Normalization

  • Eliminates Data Redundancy: Reduces storage requirements and inconsistencies
  • Improves Data Integrity: Changes need to be made in only one place
  • Reduces Update Anomalies: Prevents inconsistent updates across multiple records
  • Simplifies Maintenance: Easier to modify database structure
  • Better Organization: Each table represents a single entity or relationship
  • Enforces Business Rules: Database structure reflects business logic

‚ö†Ô∏è Potential Drawbacks

  • Increased Complexity: More tables and relationships to manage
  • Performance Impact: May require more JOINs in queries
  • Over-normalization: Too many tables can complicate simple operations

Database Design Best Practices

üìã Planning Phase
  • Understand business requirements thoroughly
  • Identify all entities and their relationships
  • Define data types and constraints
  • Plan for future scalability
üéØ Implementation Phase
  • Follow naming conventions consistently
  • Use appropriate data types
  • Implement proper indexing strategy
  • Set up referential integrity constraints
üîß Maintenance Phase
  • Regular performance monitoring
  • Database backup and recovery procedures
  • Security audits and updates
  • Documentation maintenance

Performance Optimization Tips

‚ö° Query Optimization

  • Use Indexes Wisely: Create indexes on frequently queried columns
  • Optimize WHERE Clauses: Use selective conditions early in queries
  • Avoid SELECT *: Only select needed columns
  • Use JOINs Properly: Prefer JOINs over subqueries when possible
  • Limit Result Sets: Use TOP, LIMIT, or pagination
  • Consider Query Execution Plans: Analyze and optimize slow queries
-- Good: Selective WHERE clause SELECT CustomerID, CustomerName FROM Customers WHERE City = 'London' AND Country = 'UK'; -- Good: Using appropriate JOIN SELECT c.CustomerName, o.OrderDate FROM Customers c INNER JOIN Orders o ON c.CustomerID = o.CustomerID WHERE o.OrderDate >= '2023-01-01'; -- Good: Using indexes effectively CREATE INDEX IX_Orders_CustomerID_Date ON Orders(CustomerID, OrderDate);

Data Security Considerations

üîê Access Control
  • Implement role-based access control (RBAC)
  • Use principle of least privilege
  • Regular access reviews and audits
  • Strong authentication mechanisms
üõ°Ô∏è Data Protection
  • Encrypt sensitive data at rest and in transit
  • Implement data masking for non-production environments
  • Regular security updates and patches
  • Monitor and log database activities
üíæ Backup & Recovery
  • Regular automated backups
  • Test recovery procedures regularly
  • Implement point-in-time recovery
  • Off-site backup storage
