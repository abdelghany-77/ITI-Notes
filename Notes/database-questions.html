Part I: Database Theory & Concepts
Q1
The normal forms 1NF and 2NF are intermediate steps to 3NF.
‚úì TRUE
FALSE

Correct Answer: TRUE

To achieve 3NF, a relation must first be in 1NF, then 2NF. Each normal form builds upon the previous one as progressive steps toward better database design.

Q2
A relation R is in the second normal form if and only if it is in 1NF and every nonkey attribute is fully dependent on the foreign key.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: A relation is in 2NF if it is in 1NF and every nonkey attribute is fully dependent on the primary key, not the foreign key.

Q3
Database management system is a layer of software between the physical database itself and the users of the system.
‚úì TRUE
FALSE

Correct Answer: TRUE

DBMS acts as an interface layer that manages data storage, retrieval, and manipulation while abstracting the physical implementation from users.

Q4
SQL language has data definition statements such as INSERT, UPDATE, DELETE and data manipulation statements such as CREATE.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: DDL (Data Definition Language) includes CREATE, ALTER, DROP. DML (Data Manipulation Language) includes INSERT, UPDATE, DELETE.

Q5

TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: The relationship between Employee and Project is typically many-to-many (M:N) because an employee can work on multiple projects and a project can have multiple employees.

Q6
Two distinct tuples cannot have identical values in all fields that comprise a primary key.
‚úì TRUE
FALSE

Correct Answer: TRUE

Primary key constraint ensures uniqueness - no two rows can have the same primary key value(s).

Q8
If a base table is dropped, all the related views will be automatically dropped.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: Views are not automatically dropped when base tables are dropped. The views become invalid but remain in the system catalog.

Q9
There are physical records corresponding to user created views.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: Views are virtual tables - they don't store data physically. The data is retrieved from underlying base tables when the view is queried.

Q10
Relational database relation may accept multi-valued data items (attributes).
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: In relational databases, each attribute must contain atomic (single) values. Multi-valued attributes violate 1NF.

Q11
Deleting all tuples of a relation is equivalent to dropping it.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: Deleting all tuples (DELETE command) removes data but keeps the table structure. Dropping (DROP command) removes both data and structure.

Q12
Part of the primary key may accept null value.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: Primary key attributes cannot contain NULL values. All components of a primary key must be NOT NULL.

Q13
In a relation the tuples are ordered top to bottom.
TRUE
‚úì FALSE

Correct Answer: FALSE

Correction: In relational theory, tuples in a relation are unordered. Order is only imposed when using ORDER BY in queries.

Q14
Which of the following is true about the following SQL statement?
SELECT * FROM products WHERE quantity = 1 or quantity = 2 and colour='red'
A) Product with quantity 1 and any colour
B) Product with quantity 2 and any colour
C) Product with quantity 2 and colour=red
D) A,B
E) A,C

Correct Answer: E) A,C

Due to operator precedence, AND is evaluated before OR. The query returns: (quantity = 1) OR (quantity = 2 AND colour='red')

Q15

Functional dependency X ‚Üí Y means that for each value of X, there is exactly one corresponding value of Y.

Q16

Q18

C) SELECT name, population FROM country GROUP BY continent

Part III: Database Design & Normalization
Q27
Given the table:

Donation (DonorID#, Date, Name, Member, Phone, Address, City, State, Zip, Location, Description, Age, Price, Quantity, Value, TotalValue)

With DonorID# and Date as primary key, normalize this table to 3NF.

Normalization Solution:

1NF to 2NF: Remove partial dependencies

Donor (DonorID#, Name, Member, Phone, Address, City, State, Zip, Location, Age) Donation (DonorID#, Date) DonationItem (DonorID#, Date, Description, Price, Quantity, Value)

2NF to 3NF: Remove transitive dependencies

Donor (DonorID#, Name, Member, Phone, Age, LocationID) Location (LocationID, Address, City, State, Zip) Donation (DonorID#, Date) DonationItem (DonorID#, Date, Description, Price, Quantity, Value)
Q28
For a banking system database, design ER diagram with entities: Customer, Branch, Credit, Transaction. Include all relationships and constraints.

ER Diagram Components:

Entities:

  • Customer (cust_id, name, address, phone)
  • Branch (branch_name, address, phone, loan_permitted)
  • Credit (credit_no, amount, last_transaction_date)
  • Transaction (trns_no, trans_type, trans_date, trans_amount)

Relationships:

  • Customer-Credit: 1:1 (each customer has exactly one credit)
  • Customer-Transaction: 1:M (customer can have many transactions)
  • Branch-Transaction: 1:M (branch processes many transactions)
  • Customer-Branch: M:N (customer can deal with any branch)

Normalized Relations:

This design satisfies all the given constraints and maintains referential integrity.

Q30
Write SQL to display pilot names who can operate planes with cruising range > 3000 miles but are not certified on any Boeing aircraft.

SQL Solution:

SELECT DISTINCT p.ename FROM Pilots p JOIN Certified c ON p.eid = c.eid JOIN Aircraft a ON c.aid = a.aid WHERE a.cruisingrange > 3000 AND p.eid NOT IN ( SELECT c2.eid FROM Certified c2 JOIN Aircraft a2 ON c2.aid = a2.aid WHERE a2.aname LIKE '%Boeing%' );
Q31
Write SQL to show department names, locations, employees' last names, job titles and salaries for employees working in location 1800.

SQL Solution:

SELECT d.dept_name, d.location_id, e.lname, e.job_name, e.salary FROM Departments d JOIN Employees e ON d.dept_id = e.dept_id WHERE d.location_id = 1800;
Q32
Write SQL to show department names, locations and number of employees, including departments without employees.

SQL Solution:

SELECT d.dept_name, d.location_id, COUNT(e.emp_id) as emp_count FROM Departments d LEFT JOIN Employees e ON d.dept_id = e.dept_id GROUP BY d.dept_name, d.location_id;
Q33
Write SQL to create view EMP_DEPT with columns ENAME, DNAME, SAL, excluding anyone with salary < 1000.

SQL Solution:

CREATE VIEW EMP_DEPT AS SELECT e.ename, d.dname, e.sal FROM Employee e JOIN Department d ON e.deptno = d.deptno WHERE e.sal >= 1000; -- To drop the view: DROP VIEW EMP_DEPT;
Q34
Write SQL to find employees who worked (not as project manager) in more than one project.

SQL Solution:

SELECT e.ename FROM Employee e JOIN Employee_team et ON e.empno = et.empno JOIN Project p ON et.team_id = p.team_id JOIN Team t ON p.team_id = t.team_id WHERE e.empno != t.mgrno GROUP BY e.empno, e.ename HAVING COUNT(DISTINCT p.proj_id) > 1;
