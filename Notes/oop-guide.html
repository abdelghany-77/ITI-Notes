🔍 Overview

Object-Oriented Programming (OOP) is a programming paradigm that uses objects and classes to structure code. PHP supports OOP, allowing developers to write modular, reusable, and maintainable code.

🎯 Key Concepts

  1. Class: A blueprint for creating objects.
  2. Object: An instance of a class.
  3. Properties: Variables that belong to a class.
  4. Methods: Functions that belong to a class.
  5. Encapsulation: Restricting access to certain properties or methods.
  6. Inheritance: Creating a new class from an existing class.
  7. Polymorphism: Using a single interface to represent different types.
  8. Abstraction: Hiding complex implementation details.

💻 Basic Syntax

Defining a Class

class ClassName {
    // Properties
    public $property1;
    private $property2;
    
    // Methods
    public function method1() {
        // Code
    }
    
    private function method2() {
        // Code
    }
}

Creating an Object

$object = new ClassName();

Accessing Properties and Methods

$object->property1 = "value";
$object->method1();

🏗️ Properties and Methods

Visibility Modifiers

  • Public: Accessible from anywhere.
  • Private: Accessible only within the class.
  • Protected: Accessible within the class and its subclasses.

Example:

class Car {
    public $model;
    private $mileage;
    protected $color;

    public function setMileage($mileage) {
        $this->mileage = $mileage;
    }

    public function getMileage() {
        return $this->mileage;
    }
}

Static Properties and Methods

  • Belong to the class, not an instance.
  • Accessed using ::.

Example:

class Math {
    public static $pi = 3.14;

    public static function square($num) {
        return $num * $num;
    }
}

echo Math::$pi; // 3.14
echo Math::square(5); // 25

⚙️ Constructor and Destructor

Constructor

  • Called when an object is created.
  • Use __construct().

Example:

class Car {
    public $model;

    public function __construct($model) {
        $this->model = $model;
    }
}

$car = new Car("Toyota");
echo $car->model; // Toyota

Destructor

  • Called when an object is destroyed.
  • Use __destruct().

Example:

class Car {
    public function __destruct() {
        echo "Car object destroyed.";
    }
}

$car = new Car();
unset($car); // Car object destroyed.

🌳 Inheritance

  • A class can inherit properties and methods from another class.
  • Use extends.

Example:

class Vehicle {
    public $brand;

    public function honk() {
        return "Honk!";
    }
}

class Car extends Vehicle {
    public function drive() {
        return "Driving a $this->brand car.";
    }
}

$car = new Car();
$car->brand = "Toyota";
echo $car->honk(); // Honk!
echo $car->drive(); // Driving a Toyota car.

🔄 Method Overriding

A subclass can override a method from its parent class.

Example:

class Vehicle {
    public function honk() {
        return "Honk!";
    }
}

class Car extends Vehicle {
    public function honk() {
        return "Beep!";
    }
}

$car = new Car();
echo $car->honk(); // Beep!

🔒 Abstract Classes

  • Cannot be instantiated.
  • Used as a base class for other classes.
  • Use abstract.

Example:

abstract class Animal {
    abstract public function makeSound();
}

class Dog extends Animal {
    public function makeSound() {
        return "Woof!";
    }
}

$dog = new Dog();
echo $dog->makeSound(); // Woof!

🤝 Interfaces

  • Defines a contract for classes.
  • Use interface and implements.

Example:

interface Sound {
    public function makeSound();
}

class Dog implements Sound {
    public function makeSound() {
        return "Woof!";
    }
}

$dog = new Dog();
echo $dog->makeSound(); // Woof!

🔗 Traits

  • Allow code reuse in single inheritance.
  • Use trait.

Example:

trait Loggable {
    public function log($message) {
        echo "Log: $message";
    }
}

class User {
    use Loggable;

    public function save() {
        $this->log("User saved.");
    }
}

$user = new User();
$user->save(); // Log: User saved.

✨ Magic Methods

  • Special methods that start with __.
  • Examples: __construct(), __destruct(), __get(), __set(), __toString().

Example:

class Car {
    private $data = [];

    public function __set($name, $value) {
        $this->data[$name] = $value;
    }

    public function __get($name) {
        return $this->data[$name];
    }

    public function __toString() {
        return "This is a car object.";
    }
}

$car = new Car();
$car->model = "Toyota";
echo $car->model; // Toyota
echo $car; // This is a car object.

📁 Namespaces

  • Prevent name conflicts between classes.
  • Use namespace.

Example:

namespace App;

class User {
    public function greet() {
        return "Hello!";
    }
}

$user = new \App\User();
echo $user->greet(); // Hello!

🔄 Autoloading

  • Automatically load classes without require or include.
  • Use spl_autoload_register().

Example:

spl_autoload_register(function ($class) {
    include 'classes/' . $class . '.php';
});

$user = new User();

📝 Complete OOP Workflow Example

1. Define a Class:

class User {
    private $name;

    public function __construct($name) {
        $this->name = $name;
    }

    public function greet() {
        return "Hello, $this->name!";
    }
}

2. Create an Object:

$user = new User("John");
echo $user->greet(); // Hello, John!

3. Extend a Class:

class Admin extends User {
    public function promote() {
        return "$this->name has been promoted to admin.";
    }
}

$admin = new Admin("Jane");
echo $admin->greet(); // Hello, Jane!
echo $admin->promote(); // Jane has been promoted to admin.

🧱 SOLID Principles

SOLID is an acronym for five design principles introduced by Robert C. Martin (Uncle Bob). These principles help developers design software that is easy to understand, maintain, and extend.

1. Single Responsibility Principle (SRP)

Definition:

A class should have only one reason to change, meaning it should have only one job or responsibility.

Why:

  • Improves readability and maintainability.
  • Reduces the risk of introducing bugs when making changes.

Violation Example:

class User {
    public function authenticate($username, $password) {
        // Authentication logic
    }

    public function logError($error) {
        // Logging logic
    }
}

Solution:

class User {
    public function authenticate($username, $password) {
        // Authentication logic
    }
}

class Logger {
    public function logError($error) {
        // Logging logic
    }
}

2. Open/Closed Principle (OCP)

Definition:

Software entities (classes, modules, functions) should be open for extension but closed for modification.

Why:

  • Allows adding new features without changing existing code.
  • Reduces the risk of breaking existing functionality.

Violation Example:

class AreaCalculator {
    public function calculate($shape) {
        if ($shape instanceof Circle) {
            return pi() * $shape->radius ** 2;
        } elseif ($shape instanceof Square) {
            return $shape->side ** 2;
        }
    }
}

Solution:

interface Shape {
    public function area();
}

class Circle implements Shape {
    public $radius;

    public function area() {
        return pi() * $this->radius ** 2;
    }
}

class Square implements Shape {
    public $side;

    public function area() {
        return $this->side ** 2;
    }
}

class AreaCalculator {
    public function calculate(Shape $shape) {
        return $shape->area();
    }
}

3. Liskov Substitution Principle (LSP)

Definition:

Objects of a superclass should be replaceable with objects of a subclass without affecting the correctness of the program.

Why:

  • Ensures a subclass can stand in for its superclass without causing unexpected behavior.
  • Promotes proper inheritance.

Violation Example:

class Bird {
    public function fly() {
        // Fly logic
    }
}

class Ostrich extends Bird {
    public function fly() {
        throw new Exception("Can't fly");
    }
}

Solution:

class Bird {
    // Common bird methods
}

class FlyingBird extends Bird {
    public function fly() {
        // Fly logic
    }
}

class Ostrich extends Bird {
    // Ostrich-specific methods
}

4. Interface Segregation Principle (ISP)

Definition:

Clients should not be forced to depend on interfaces they do not use. Instead of one large interface, create smaller, specific interfaces.

Why:

  • Reduces the burden of implementing unnecessary methods.
  • Makes the system more modular and easier to refactor.

Violation Example:

interface Worker {
    public function work();
    public function eat();
}

class Human implements Worker {
    public function work() {
        // Work logic
    }

    public function eat() {
        // Eat logic
    }
}

class Robot implements Worker {
    public function work() {
        // Work logic
    }

    public function eat() {
        // Robots don't eat!
    }
}

Solution:

interface Workable {
    public function work();
}

interface Eatable {
    public function eat();
}

class Human implements Workable, Eatable {
    public function work() {
        // Work logic
    }

    public function eat() {
        // Eat logic
    }
}

class Robot implements Workable {
    public function work() {
        // Work logic
    }
}

5. Dependency Inversion Principle (DIP)

Definition:

High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

Why:

  • Promotes loose coupling between modules.
  • Makes the system more flexible and easier to test.

Violation Example:

class MySQLConnection {
    public function connect() {
        // MySQL connection logic
    }
}

class User {
    private $dbConnection;

    public function __construct() {
        $this->dbConnection = new MySQLConnection();
    }
}

Solution:

interface DBConnection {
    public function connect();
}

class MySQLConnection implements DBConnection {
    public function connect() {
        // MySQL connection logic
    }
}

class User {
    private $dbConnection;

    public function __construct(DBConnection $dbConnection) {
        $this->dbConnection = $dbConnection;
    }
}

Summary of SOLID Principles

Principle Key Idea Example
Single Responsibility (SRP) A class should have only one reason to change. Separate authentication and logging into different classes.
Open/Closed (OCP) Classes should be open for extension but closed for modification. Use interfaces to allow adding new shapes without modifying existing code.
Liskov Substitution (LSP) Subclasses should be substitutable for their base classes. Ensure subclasses can replace superclasses without breaking the program.
Interface Segregation (ISP) Create small, specific interfaces instead of large, general-purpose ones. Split a large Worker interface into Workable and Eatable.
Dependency Inversion (DIP) Depends on abstractions, not concretions. Inject database connections via an interface instead of hardcoding them.

🏗️ Design Patterns

Design patterns are reusable solutions to common problems in software design. They represent best practices and provide a template for how to solve problems that occur repeatedly in software development.

🔧 Creational Patterns

Deal with object creation mechanisms

1. Singleton Pattern

PHP
class Database {
    private static $instance = null;
    private $connection;
    
    private function __construct() {
        // Private constructor prevents instantiation
        $this->connection = new PDO('mysql:host=localhost');
    }
    
    public static function getInstance() {
        if (self::$instance === null) {
            self::$instance = new self();
        }
        return self::$instance;
    }
    
    private function __clone() {} // Prevent cloning
    private function __wakeup() {} // Prevent unserialization
}

2. Factory Pattern

PHP
interface Logger {
    public function log($message);
}

class FileLogger implements Logger {
    public function log($message) {
        file_put_contents('app.log', $message, FILE_APPEND);
    }
}

class DatabaseLogger implements Logger {
    public function log($message) {
        // Log to database
    }
}

class LoggerFactory {
    public static function create($type) {
        switch ($type) {
            case 'file':
                return new FileLogger();
            case 'database':
                return new DatabaseLogger();
            default:
                throw new InvalidArgumentException('Unknown logger type');
        }
    }
}

🏗️ Structural Patterns

Deal with object composition and relationships

3. Adapter Pattern

PHP
// Old payment system
class OldPaymentSystem {
    public function makePayment($amount) {
        return "Payment of $" . $amount . " processed";
    }
}

// New payment interface
interface PaymentInterface {
    public function pay($amount);
}

// Adapter to make old system work with new interface
class PaymentAdapter implements PaymentInterface {
    private $oldPaymentSystem;
    
    public function __construct(OldPaymentSystem $oldPaymentSystem) {
        $this->oldPaymentSystem = $oldPaymentSystem;
    }
    
    public function pay($amount) {
        return $this->oldPaymentSystem->makePayment($amount);
    }
}

⚙️ Behavioral Patterns

Deal with communication between objects and responsibilities

4. Observer Pattern

PHP
interface Observer {
    public function update($data);
}

class EmailNotifier implements Observer {
    public function update($data) {
        echo "Email sent: " . $data;
    }
}

class UserRegistration {
    private $observers = [];
    
    public function addObserver(Observer $observer) {
        $this->observers[] = $observer;
    }
    
    public function register($user) {
        // Register user logic...
        $this->notifyObservers("User " . $user . " registered");
    }
    
    private function notifyObservers($data) {
        foreach ($this->observers as $observer) {
            $observer->update($data);
        }
    }
}

⭐ Best Practices

Following OOP best practices helps create maintainable, readable, and robust code. Here are essential guidelines for effective OOP development.

📝 Naming Conventions

  • Classes: PascalCase (e.g., UserAccount)
  • Methods: camelCase (e.g., getUserName())
  • Properties: camelCase (e.g., $firstName)
  • Constants: UPPER_CASE (e.g., MAX_ATTEMPTS)

🔒 Encapsulation Rules

  • Keep properties private/protected
  • Provide public getters/setters when needed
  • Validate data in setters
  • Hide internal implementation details

🏗️ Class Design

  • Keep classes focused (Single Responsibility)
  • Favor composition over inheritance
  • Use interfaces for contracts
  • Make classes immutable when possible

📚 Code Organization

  • Use namespaces to organize code
  • Follow PSR-4 autoloading standards
  • Group related classes in directories
  • Use meaningful file and directory names

💡 Common Anti-Patterns to Avoid

❌ Bad Example
// God Object - does too much
class User {
    public function validateEmail() { /* ... */ }
    public function hashPassword() { /* ... */ }
    public function sendEmail() { /* ... */ }
    public function logActivity() { /* ... */ }
    public function generateReport() { /* ... */ }
    // Too many responsibilities!
}
✅ Good Example
// Single Responsibility - focused classes
class User {
    private $email;
    private $password;
    
    // Only user-related data and basic operations
}

class EmailValidator {
    public function validate($email) { /* ... */ }
}

class PasswordHasher {
    public function hash($password) { /* ... */ }
}

class EmailService {
    public function send($to, $message) { /* ... */ }
}

🧪 Testing Best Practices

  • Unit Testing: Test individual methods in isolation
  • Mock Dependencies: Use dependency injection for testability
  • Test Coverage: Aim for high test coverage
  • TDD: Consider Test-Driven Development approach

🚀 Performance Considerations

  • Lazy Loading: Load objects only when needed
  • Object Pooling: Reuse expensive objects
  • Caching: Cache frequently accessed data
  • Memory Management: Unset unused object references

PHP
/**
 * Represents a bank account with basic operations
 * 
 * @package Banking
 * @author Your Name
 * @version 1.0
 */
class BankAccount {
    /**
     * Account balance in cents to avoid floating point issues
     * 
     * @var int
     */
    private $balanceInCents;
    
    /**
     * Deposit money into the account
     * 
     * @param float $amount Amount to deposit
     * @return bool True if successful
     * @throws InvalidArgumentException If amount is negative
     */
    public function deposit($amount) {
        if ($amount <= 0) {
            throw new InvalidArgumentException('Amount must be positive');
        }
        
        $this->balanceInCents += intval($amount * 100);
        return true;
    }
}
